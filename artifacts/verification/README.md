# Verification artifacts

Files in this directory are command-generated release evidence. Do not hand-edit
them. Regenerate them with the generator named inside each JSON object, then run
`./scripts/generate-claim-map.sh` so `docs/CLAIM_MAP.json` captures the exact new
SHA-256.

`PASS` applies only to a completed local check. External gates use explicit
states: `DEMO_READY` or `DEMO_READINESS_BLOCKED`, and `VERIFIED` or `BLOCKED`.
GitHub's API-only gate uses `API_VERIFIED` or `BLOCKED`; it never substitutes
for the separately required authenticated-browser About readback.
Missing OpenAI keys, credentials, DNS, reachable hosting, monitoring, or GitHub
metadata never become a pass and never create substitute records.

The full local evidence set is generated by:

```bash
./scripts/verify-release.sh --strict
```

Strict verification uses a unique per-run Compose project with fresh disposable
volumes, in-memory high-entropy secrets, and loopback-only per-run Caddy ports.
The verifier captures deployment and public-key readback while that release-mode
stack is healthy, then performs project-identity-guarded cleanup of only its own
containers and volumes. Those artifacts are point-in-time provenance; they do
not assert that the disposable stack remains live after verification and do not
establish hosted availability or persistence. Runtime secrets are not written to
`.env`, command arguments, logs, or artifacts.

Supply-chain generators write the SBOM and secret scan before any Compose build,
then write image and public-key provenance after a strict deployment. A failed
preflight prevents the build:

- `artifacts/verification/source-integrity.json` proves the build source matched
  Git `HEAD`, permitting only Docker-ignored generator-owned evidence changes;
- `artifacts/verification/sbom.cdx.json` from `uv.lock` and
  `package-lock.json`;
- `artifacts/verification/build-context-secret-scan.json` from the effective
  root `.dockerignore` context;
- `artifacts/verification/immutable-build.json` from the single preflight-bound
  tar snapshot streamed to all local image builds;
- `artifacts/verification/image-provenance.json` from strict deployed Compose
  container/image readback, mandatory context labels, and an identical
  post-build context rescan; and
- `artifacts/verification/export-public-key.json`, which is distributable public
  Ed25519 verification material proven by the running API and never the private
  seed.

`artifacts/verification/claim-provenance.json` validates the registered claim
inputs, allowed states, hashes, and executable generator identities before the
claim map is published. `artifacts/verification/warrant-boundary.json` includes
the live isolated production Broker-to-runner probe as well as spoof, nonce,
PostgreSQL, candidate-sidecar, and MCP boundary checks.

A provider-shaped test fixture must live under `tests/fixtures/` and requires the
same-line annotation `# secret-scan: allow=test-fixture`; the scanner records the
allowance without recording the value. The marker is ignored outside that tree.
Never annotate real credentials. Image provenance and the public-key artifact
are strict gates: missing Docker state or export seed makes the release fail.
Blocked stages overwrite stale PASS artifacts. Their local success does not
establish hosted acceptance.

Genuine model runs, if credentials exist, are written as JSON Lines plus one
content-addressed incident artifact per run. Raw evidence and credentials must
never be copied here.
